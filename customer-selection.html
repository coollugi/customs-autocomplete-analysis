2026-04-20

涵蓋範圍：選納稅義務人 / 選買賣方 / 選收貨人 / 選付款人 / 客戶料號帶入 + 連鎖觸發

背景：報單表頭裡「客戶」分四個角色 — 納稅義務人、買賣方、收貨人、付款人。OP 在表頭輸入客戶代碼後，LRP 會從客戶主檔（CUSTOM）+ 客戶部門設定（CUSTDEP）+ 客戶聯絡人（CUSTOMDA）一次拉出十幾個欄位寫進報單；項次層的客戶料號（PLMIDCC）則是 OP 輸入「客戶料號」後一口氣帶入貨名 / 稅則 / 品牌 / 統計方式 / 單價等十多個欄位。
本份報告把這 4 個事件 + 1 個項次事件實際做了什麼、從哪幾張表拉、連鎖觸發了什麼，全部攤開給五崧確認。

5.3 連鎖觸發 — 稅則重查

  1. 進入 S02Exit 前先記下當前 CCC_CODE（ccc_code := s07.Field.AsString）
  2. Read_LH 帶入新的 CCC_CODE
  3. 若 s07.Field.AsString ≠ ccc_code，呼叫 S07Exit(S07) → 重新查 PLMCCC 帶入：稅率（從價/從量）/ 統計單位 / 簽證代號（IM_PMT1~7）
連鎖效果：選一個客戶料號等於一次帶入：13 行貨名 + 稅則 + 商品分類 + 品牌 + 統計方式 + 單價 + 生產國別 + 數量單位 + （連鎖）稅率/簽證共 19+ 個欄位。完整稅則連鎖規則詳見包 4。

5.4 貨名顯示處理

Read_LH 帶完欄位後，會呼叫 Set_GoodsNameToTStrings 把 DESCP1~13 + SCUSBLOB 合併成一段文字寫到 Memo1.Lines（畫面上「貨物名稱」大區塊），讓 OP 看到完整貨名而不是 13 個獨立欄位。

一、聯絡人選取 UX 改善

LRP 現行做法

選納稅義務人後，Get_CUSTOMDA 撈出該客戶的所有「一般聯絡人」（add_index NULL 或 'A'），多筆就跳出 F1_LOOKUP 視窗讓 OP 點選一筆。

DB 實證：35,775 筆聯絡人覆蓋 7,488 客戶，平均每客戶 4.8 筆 — 跳視窗是常態，不是例外。

「主要聯絡人」（main_contact='Y'）只有 4.6% 設定，且非機場轉運（CT）業務不看這個旗標。

新系統建議做法

聯絡人選取走 「主要聯絡人優先 → fallback 跳視窗」 邏輯：

  1. 客戶有 main_contact='Y' → 直接帶入，不打擾 OP
  2. 客戶有 1 筆聯絡人 → 直接帶入
  3. 客戶有 N 筆聯絡人且無主要 → 跳視窗（依舊行為）

同時：把「main_contact='Y'」的設定從 4.6% 推升到接近 100% — 提供管理介面批次標記。

效益估算：跳視窗的減少量取決於「主要聯絡人」標記的覆蓋率。
• 現況 main_contact='Y' 覆蓋率 4.6% → 直接套新邏輯只能消除 4.6% 的跳視窗操作
• 若客戶部門配合批次標記，把覆蓋率推到 80%+ → 才能達到對應比例的消除效果

二、cust_degree 欄位的去留決策

事實：CUSTDEP.cust_degree（信任等級）在 6,264 筆中無一筆有資料，但 LRP 仍把它寫到報單 filer1 欄位 — 等於每張報單 filer1 都帶空字串。

2.1 三種可能的處置方向

方向適用情境新系統做法
移除確認業務不再使用「客戶信任等級」概念客戶主檔不設此欄位、報單 filer1 也不再有此來源
保留但標記為廢棄不確定是否還會用保留欄位但不在 UI 出現、不參與 AutoComplete
復活業務希望分等級管理客戶但 LRP 沒做好主檔提供下拉選單、報關時依等級套不同流程（例如 VIP 客戶先審查）

三、繳款人重算的時機顯式化

LRP 現行做法

選納稅義務人 / 選買方 後都會自動呼叫 Set_DeclPayer_Local，跑完整 7 級瀑布配對 — OP 無感、不知道剛才繳款人改了。

包 3 會詳細說明 7 級瀑布規則。這裡的問題是：OP 沒辦法看到繳款人為什麼變了。

新系統建議做法

繳款人欄位旁邊加一個小提示：「來自 [規則名]」，例如：

  • 「來自常態配對表 #042」
  • 「來自快遞配對表」
  • 「無配對 → 預設為納稅義務人」
  • 「OP 手動指定（已鎖定）」

讓 OP 知道每次系統自動換繳款人的原因，而不是黑箱。

四、客戶料號的「進出口分流」資料模型決策

4.1 LRP 現況

PLMIDCC 用 3 個 key：(custom_no, in_exp, id_no) — 同一個料號代碼可以在進口（'I'）和出口（'E'）各存一筆，內容（稅則/單價/品牌）可以不同。

DB 實證：248,040 筆進口 + 204,071 筆出口 = 45.2 萬。Top 客戶 B010 進口 10.4 萬 + 出口 10.2 萬 — 進出口幾乎對半。

4.2 兩個資料模型選擇

方案 A：沿用三 key 模型

客戶料號表保留 in_exp 欄位作為 key 一部分。

優點：與 LRP 完全相容、資料可直接遷移。

缺點：同代碼需維護兩筆、貨名重複、容易不一致。

方案 B：合併為單一料號

客戶料號表用 (custom_no, id_no) 兩 key。in_exp 改成料號內的標籤陣列（這個料號可進口、可出口、或兩者都可）。

進出口屬性差異（如 P_CTYCODE 只進口用）改為「在報單上條件帶入」。

優點：資料模型乾淨、貨名一致。

缺點：需評估真實進出口差異有多大、資料遷移要做合併判斷。

判斷標準：實務上同代碼進出口料號內容差異很小 → 用方案 B 簡化；差異大（例如進出口稅則完全不同）→ 用方案 A。

五、客戶料號維護介面現代化

LRP 現行做法

料號維護是 Delphi 表單，一筆一筆編輯。Top 1 客戶 B010 維護 20.5 萬筆 — 表單操作幾乎不可能，應該已有批次匯入工具，但散在 LRP 各處。

新系統建議做法

  • 客戶側匯入介面：客戶上傳 CSV/Excel 自動更新料號（含驗證）
  • 差異比對：上傳前顯示「新增 X 筆 / 更新 Y 筆 / 衝突 Z 筆」
  • 稅則對照建議：稅則欄位空時，依貨名 fuzzy match 推薦稅則
  • 歷史單價統計：lu_price 從歷史報單統計，提供「過去 N 次平均」參考
數據驅動的依據：CCC_CODE 填寫率 55.4% / LU_PRICE 填寫率 43.3% — 過半料號沒設稅則或單價。新系統提供「歷史學習 + 自動建議」功能可顯著提升填寫率。

六、需確認後再優化的條件邏輯

6.1 SHIPTO_CODEEQ（收貨人驗證號碼）

選收貨人時若 codeq 非空就帶入 SHIPTO_CODEEQ — 用途不明，需先釐清業務情境再決定新系統怎麼做。

6.2 運輸方式 99 → 自動加包裝描述

選買賣方時，若運輸方式 = 99（國內），會把客戶 pro_no 追加到「其他申報事項」欄位。這個條件邏輯散在多處，可選：

  • 合併為單一規則：「99 國內運輸 → 套用 XX 模板」
  • 或者拆成獨立的「報單模板」概念，由 OP 主動套用

6.3 「找不到客戶」的 fallback

買方/賣方找不到時自動把 OP 輸入字串當外部統編 — 海外公司常見。新系統建議：

  • UI 明確區分「客戶代碼」和「外部統編」兩種輸入
  • 外部統編直接寫入欄位，不嘗試查 CUSTOM 表
  • 客戶代碼找不到時顯示明確錯誤訊息，不要靜默 fallback

七、新系統建議新增的功能

7.1 客戶角色一鍵帶入

實務上很多報單的「納稅義務人 = 買方/賣方 = 收貨人」（同一家公司）。新系統可加「同上」按鈕，一鍵把納稅義務人複製到買方/收貨人，省 OP 三次輸入。

7.2 客戶料號智慧推薦

OP 輸入貨名時，依字串相似度推薦該客戶歷史料號（前 5 筆），減少 OP 記憶負擔。

7.3 料號帶入後的「為什麼這個值」

料號帶入後，每個欄位旁顯示來源（料號設定 / 歷史報單統計 / 系統預設），讓 OP 知道哪些值是「料號設定」可信、哪些是「猜的」需要驗證。

7.4 客戶料號活躍度指標

對 OP 顯示「這個客戶料號最後一次使用：N 天前 / 共使用 M 次」— 老料號的稅則 / 單價可能要刷新。

7.5 連絡人主要標記批次設定工具

提供管理工具，讓客戶部門可批次標記每個客戶的「主要聯絡人」 — 把 4.6% 推到 100%，配合第一項的 UX 改善。

核心原則：業務邏輯全部保留，OP 操作介面則大幅優化 — 多筆聯絡人不要 force 跳視窗、料號維護從 Delphi 表單改為批次匯入、繳款人變動原因要可見。
