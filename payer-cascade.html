三、前置處理（4 項自動轉換）

進入 7 級瀑布前，系統會先做幾項欄位轉換，這些轉換對 OP 來說透明，但會影響後續配對結果。

3.1 CUSTOM_NO ↔ BS_NO 自動反轉

當報單為轉運（DEP=FT）或進倉退關（DECL_TYPE in G2/B1）時，系統會把納稅義務人與交易對象互換後再進入瀑布判斷。

條件反轉前反轉後（用來查 custpayer）
DEP = FTCUSTOM_NO=A / BS_NO=B當作 CUSTOM_NO=B / BS_NO=A 查詢
DECL_TYPE = G2 或 B1CUSTOM_NO=A / BS_NO=B當作 CUSTOM_NO=B / BS_NO=A 查詢

PUB_PROC.pas L4848-4854

3.2 DHL → 洋基快遞單（CT 限定）

當為轉運報單（DEP=CT）且 TRANS_TYPE='DHL' 時，系統自動把轉運方式改成「洋基快遞單」再去查配對表。

PUB_PROC.pas L4884，註記為 2015-06-17 因新增 367 箱號而調整

四、7 級優先瀑布

系統依以下順序判斷主要付款人，前面的規則先命中就停止，不再往下找：

1
OP 手動指定（PAYER_BYUSER='Y'）
條件：OP 在報單上手動勾選或修改過付款人
→ 結果：保留現有 PAYER_ID 不變
2
快遞單付款人特殊設定命中
條件：空運公司有設定 cust_express_payer_spec，且當前報單命中其 5 維組合（DEP × DECL_TYPE × SCIAREA_YN × TRANS_TYPE × DEC_CUSTOM_NO）
→ 結果：付款人 = spec.PAYER_ID（若為 'VR_EXP' 則替換為當前報單的 CUSTOM_NO）
3
FreeHouse / EXW 出口
條件：勾選 FREEHOUSE_YN='Y'，或 TERMS='EXW' 且 DEP in (CE/FE) 且空運公司有值，且 TPT_MODE ≠ '42'
→ 結果：付款人 = ACRY_CMPID（空運／承攬公司）
4
合作外銷（CORP='Y'）
條件：勾選合作外銷
→ 結果：付款人 = TX_NO（保稅廠商）
5
常態付款人命中（custpayer）
條件：以納稅義務人為 key，命中 custpayer 表的 4 維組合（DEP × DECL_TYPE × SCIAREA_YN × TPT_MODE）+ 次要條件（BS_NO / ACRY_CMPID / TRANS_TYPE）
→ 結果：付款人 = custpayer.PAYER_ID
6
快遞單付款人命中（cust_express_payer）
條件：以空運公司為 key，且該空運公司在快遞單付款人清單中是有資料的成員，命中 3 維組合（DEP × DECL_TYPE × SCIAREA_YN）+ 次要條件（TRANS_TYPE，僅 CT 比對）
→ 結果：付款人 = cust_express_payer.PAYER_ID
7
Fallback：付款人 = 納稅義務人本人
條件：以上六項皆未命中
→ 結果：付款人 = CUSTOM_NO（納稅義務人本人）

七、次要條件比對

fallback 矩陣命中後，系統會再用「次要條件」驗證該筆規則是否真的適用本報單。次要條件不參與 fallback，是「該筆規則的額外限制」。

7.1 custpayer 三項次要條件

欄位判斷
BS_NO（指定交易對象）若規則此欄位有值，必須等於本報單的 BS_NO 才視為命中
ACRY_CMPID（指定空運公司）若規則此欄位有值，必須等於本報單的 ACRY_CMPID 才視為命中
TRANS_TYPE（指定轉運方式）僅當 DEP=CT 時比對；若規則此欄位有值，必須等於本報單轉換後的 TRANS_TYPE 才視為命中

7.2 cust_express_payer 一項次要條件

欄位判斷
TRANS_TYPE僅當 DEP=CT 時比對；若規則此欄位有值，必須等於本報單轉換後的 TRANS_TYPE 才視為命中

7.3 cust_express_payer_spec 一項次要條件

欄位判斷
TRANS_TYPE僅當 DEP=CT 時比對；若規則此欄位有值，必須等於本報單轉換後的 TRANS_TYPE 才視為命中

注意：spec 表 5 維中本身就有 TRANS_TYPE 維度（會跑 fallback），這個「次要條件比對」是在進入 lcSeek_Detial 後額外的二次檢查（基本上重複）。

八、VR_EXP 虛擬付款人機制

spec 表的 PAYER_ID 欄位有一個特殊值 VR_EXP，並非真正的客戶代碼，而是一個虛擬標記。

機制：當 spec 表規則命中且 PAYER_ID='VR_EXP' 時，系統會把 PAYER_ID 替換為「當前報單的納稅義務人 CUSTOM_NO」。

PUB_PROC.pas L4613

8.1 為什麼需要這個機制？

spec 表的 5 維設計中，DEC_CUSTOM_NO 維度可以填萬用值 XXXXX。但若規則設「當條件命中時，付款人=任何客戶都用 X 客戶」這種寫法，無法表達「付款人=該客戶自己」這種語意。

VR_EXP 提供了一種「動態替換」的能力：規則寫一次，但實際結果依當前報單的客戶而變。

8.2 實際使用情況

目前資料庫只有 1 筆 spec 規則使用 VR_EXP（spec seq=3）：

  • 條件：洋基（82178）+ AJ100 竹科 + CI 進口 + 區內 + 洋基快遞單 + 任何客戶
  • 結果：付款人 = 該客戶自己

另兩筆（A098 / A353）為精準對應到「指定客戶=指定付款人=洋基」的硬規則。

cust_express_payer 表中有 7 筆規則的 custom_no='82227x'，但 LRP custom 客戶主檔中查無此代碼（只有 82227 永通國際物流）。所有 7 筆規則的 PAYER_ID 設為 82227（正確），代表設定者本意是針對永通。

二、TPT_MODE 統一為新碼，移除轉換層

現行 LRP 同時維護新舊兩套運輸方式碼，Set_DeclPayer 內呼叫 Trans_TPT_MODE_to_OLD 把新碼轉舊碼後查 custpayer，但條件判斷「TPT_MODE<>'42'」又比對的是新碼，新舊並存增加維護成本。

現行：新舊並存

  • SCUSDEC.TPT_MODE 已全面採新碼（例 41/42/12/99）
  • custpayer.TPT_MODE 多為舊碼（1/2/3/5）
  • 查詢時呼叫轉換函式 Trans_TPT_MODE_to_OLD
  • 條件判斷 TPT_MODE<>'42' 又用回新碼
  • 近一年 SCUSDEC 27 萬筆 2 碼 vs 133 筆 1 碼，舊碼幾乎絕跡

建議：新系統統一新碼

  • 所有資料表 TPT_MODE 欄位統一存 2 碼新碼
  • 遷移時把 custpayer.tpt_mode 一次轉換完畢（91% 為萬用值 X，影響範圍小）
  • 移除 Trans_TPT_MODE_to_OLD 函式
  • 條件判斷統一以新碼操作

SQL 計算依據：custpayer 109 條中 99 條（91%）tpt_mode 欄位為萬用 X，僅 10 條精準到舊碼 3/4/5；按舊→新對照轉換為 41/42/43 群組（空運非快遞 / 空運快遞 / 空運自力推進）後，覆蓋程度與原本相當。

三、VR_EXP 改顯式 enum 欄位

現行 cust_express_payer_spec 用字串值 'VR_EXP' 當虛擬付款人標記，靠程式 if 比對處理。屬於 magic string，新人接手難以察覺其特殊意義。

現行：magic string

  • PAYER_ID = 'VR_EXP' 字串
  • 程式 if 判斷後動態替換為 CUSTOM_NO
  • 客戶主檔查不到 VR_EXP（資料完整性無法驗證）

建議：顯式 enum 欄位

  • 新增 payer_resolution: enum 欄位
  • 值：STATIC（用 payer_id）/ SAME_AS_DECLARANT（=當前納稅義務人）
  • 規則語意明確，不需要看程式才懂
  • 未來易擴充其他動態解析方式（例：=當前空運公司）

影響範圍：目前只有 1 筆規則使用 VR_EXP（spec seq=3），遷移成本低。

七、AC_FLAG 鎖定後的可預覽機制

現行 AC_FLAG='Y' 後 Set_DeclPayer 直接 Exit，OP 看到的付款人是「鎖定當下的快照」。若上游規則或客戶設定變更，OP 無從察覺差異。

過去一年 95.3% 的報單為已計價鎖定狀態，這個機制是為了避免價格已通報後付款人被自動改掉。但建議新系統在「鎖定狀態」下仍能預覽 if 重算的結果，供 OP 判斷是否需要解鎖重新計算。

建議行為
  • 付款人欄位在鎖定狀態旁邊顯示「重算預覽」按鈕
  • 按下後不寫資料庫，只在 UI 顯示「依現行規則應為 X 客戶」
  • 若預覽值與當前值不同，顯示差異提示（不強迫改）
  • OP 自行決定是否解鎖（DEC_RMK 改 N）並 trigger 重算

新系統處理原則 — CustomerBondedBranch 設計覆蓋

新系統採用 CustomerBondedBranch 設計：一個 customer（公司）可有多個 branch（事業編號 / 保稅分支），每個 branch 綁固定園區與站別。三組客戶代碼的對照處理如下：

  • A098 / B001 台積電（不同園區廠） → A098 = 竹科廠 branch、B001 = 南科廠 branch，各為獨立 branch，不需合併。193 條 custpayer 規則 migration script 依 branch_id 自動分流，dimensions（DEP / DECL_TYPE / SCIAREA / TPT_MODE）保持不變。
  • A647 / C001 友達光電（名稱完全相同） → 兩種可能性：(a) 兩個獨立 branch（類似 A098/B001 拆法，例如不同廠別）、(b) LRP 歷史殘留需合併為單一 branch。需五崧協助釐清歷史背景與對應園區。
  • B021 奇美電子（名稱已標 [刪除]） → 10 條規則全部 dead，遷移時直接過濾，無需 branch 對應。

SCIAREA_YN 維度可由 branch.supervisionNo 是否存在 derived（有監管編號 = 區內），custpayer 規則 dimensions 從 4 維減為 3 維。詳見《報關系統 — 報單取號優化方案》§5.4 CustomerBondedBranch entity 職責。
